# 路径遍历防御 —— 面试标准答案

## 攻击是什么

攻击者输入 `../../etc/passwd` 或 `..\..\Windows\System32\config`，利用 `../` 跳出工作目录，读取/修改系统文件。

## 防御方法对比（5 种）

| 方法 | 怎么做 | 能防住吗 | 问题 |
|------|--------|---------|------|
| **字符串过滤** | 检查输入里有没有 `../` | ❌ 防不住 | 编码绕过：`%2e%2e%2f`、`..%5c`、`....//` 都能绕过字符串匹配 |
| **只取文件名** | `os.path.basename(path)`，丢掉目录部分 | ❌ 限制太死 | 用户没法切换到子目录，功能废了 |
| **chroot / Docker** | 把进程关在隔离的文件系统里 | ✅ 能防 | 太重，适合生产部署，不适合开发工具 |
| **白名单检查** | 直接检查用户输入是否以 `/home/user` 开头 | ❌ 不靠谱 | `../` 没消解，白名单根本匹配不上 |
| **先规范化再检查** | `realpath()` 消解 `..` → 再用白名单/黑名单 | ✅ 最佳 | 需要注意符号链接和 TOCTOU 竞态 |

## 为什么选「先规范化再检查」

### 核心原则：**不可信输入必须先消毒，再做判断**

```
用户输入 → realpath()消毒 → 黑名单 → 敏感文件 → 白名单 → 放行
          ↑ 这一步消掉所有花招
```

如果顺序反过来（先检查再消毒），等于拿脏数据做判断——`../../etc` 根本不匹配黑名单里的 `/etc`，直接漏过去了。

### realpath() 的好处

1. **消解所有 `../` 和 `./`**——不管你套多少层
2. **追踪符号链接**——软链接指向系统目录也会被识别出来
3. **返回绝对路径**——后续检查用 `startswith` 就能精确匹配
4. **标准库自带**，不依赖第三方
5. **跨平台**，Windows 的 `..\` 也能正确处理

### realpath() vs abspath()

| | `abspath()` | `realpath()` |
|---|---|---|
| 消解 `../` | ✅ | ✅ |
| 追踪符号链接 | ❌ 不追踪 | ✅ 追踪 |
| 路径必须存在 | ❌ 不要求 | ❌ 不要求（Python 3.6+） |

`realpath()` 更安全，因为攻击者可以创建指向 `/etc` 的符号链接，`abspath()` 不会发现。

## 一句话面试总结

> "路径遍历的防御核心是**先规范化再检查，顺序不能反**。用 `os.path.realpath()` 把用户输入消解成标准绝对路径后，再逐层过滤。相比于字符串过滤（容易编码绕过）和 Docker 隔离（太重），这个方法简单、有效、标准库就能做，是应用层最合适的方案。"

## 可追问的进阶点

- **TOCTOU 竞态**：检查通过后到实际操作前，文件可能被替换。Python 里用 `open(..., O_NOFOLLOW)` 缓解。
- **Windows 特殊路径**：`\\?\C:\` 前缀、NTFS 流 `file::$DATA`，需要额外处理。
- **白名单要精确匹配**：检查 `abs_path.startswith(allowed_dir + os.sep)`，不能只 `startswith(allowed_dir)`，否则 `/home/user_attack` 会通过 `/home/user` 的白名单。
