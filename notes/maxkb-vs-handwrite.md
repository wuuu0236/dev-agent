# MaxKB vs 手写 RAG 对比分析

## 一、RAG 全链路 7 步对照

| 步骤 | MaxKB 怎么做 | 你的 rag_vector.py 怎么做 | 差距 |
|------|-------------|--------------------------|------|
| ① 文档解析 | 支持 PDF/Word/Markdown/HTML/网页抓取，自动识别格式 | 只读 txt，按 `\n\n` 分段 | 多格式支持 |
| ② 文本分片 | 可配置 chunk_size、chunk_overlap、分隔符（如按标题分） | 无分片策略，直接按段落 | 检索粒度控制 |
| ③ 向量化 | 可选多种 Embedding 模型，支持切换 | 固定 `text2vec-base-chinese` | 模型可替换 |
| ④ 向量存储 | PostgreSQL + pgvector（生产级） | Chroma（本地文件） | 并发、可靠性 |
| ⑤ 检索 | 支持混合检索（关键词 + 向量 + 重排序） | 纯向量检索（余弦相似度） | 检索精度 |
| ⑥ Prompt 拼装 | 有模板引擎，支持引用标注 | 硬编码 f-string | 灵活度、可维护性 |
| ⑦ 生成回答 | 支持多模型、流式输出、对话历史 | 单次调用，无流式 | 用户体验 |

## 二、面试能讲的 3 个核心技术点

### 1. 混合检索（Hybrid Search）
纯向量检索的问题：语义相近但无关的内容也会被检索到。
解决方案：关键词检索（BM25）+ 向量检索 + Rerank 重排序。
MaxKB 用了这个策略，准确率从纯向量的 ~60% 提升到 90%+。

### 2. 分片策略对检索质量的影响
chunk_size 太大 → 检索结果包含太多噪音，LLM 抓不住重点
chunk_size 太小 → 信息不完整，LLM 看不懂上下文
chunk_overlap → 保证跨 chunk 的信息不丢失
MaxKB 默认 chunk_size=500, overlap=50，这是业界经验值。

### 3. Rerank（重排序）
向量检索返回 top_k=10 条结果 → 用 Rerank 模型二次排序 → 取最相关的 3 条喂给 LLM
好处：大幅提升回答准确率，同时控制 token 成本。

## 三、你的学习路径（从 MaxKB 反推）

1. 用 MaxKB → 理解「这个功能是干什么的」
2. 看 MaxKB 源码 → 理解「这个功能是怎么实现的」
3. 自己写简化版 → 验证「我真的会了」
4. 面试讲 → 「我部署过 MaxKB，也理解它的核心实现」
